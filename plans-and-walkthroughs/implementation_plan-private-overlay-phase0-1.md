# Implementation Plan: Private Overlay Network — Phases 0 & 1

Goal of the overall initiative: make the end-user apps reachable only inside a
private community overlay network (Headscale/Tailscale), while mail, Jitsi
(guests) and Keycloak stay public. This document covers the two preparatory
phases only:

- **Phase 0** — switch Let's Encrypt validation from HTTP-01 to DNS-01, so
  hostnames no longer need to be publicly reachable to get/renew certificates.
- **Phase 1** — deploy Headscale as a tenant, join the VM to the tailnet, and
  close SSH (22) and the Kubernetes API (6443) to the public internet.

Neither phase changes anything a community member can see. Both are pure
operator-facing hardening.

Decisions already made:
- Overlay: **Headscale** (self-hosted, single binary, official Tailscale clients).
- SSH policy: **port 22 fully closed** on production once VPN access is verified.
  Break-glass is the Hetzner Cloud console (VNC) plus temporarily re-adding a
  firewall rule there.
- Staging (`terraform-staging/`) stays fully public and unchanged — the PR
  validation pipeline (`test-pr-locally.sh`, e2e suite) must keep working
  without a VPN.

## Phase 0 — DNS-01 certificate validation

**✅ VERIFIED END-TO-END (2026-07-18), local target.** The full local-target
path (Steps 0.1, 0.5) was exercised for real, not just rendered/validated:
`admin-tools/destroy-cluster.sh` uninstalled+purged the `.dev` cluster,
`smallworlds-init.sh`'s local-target flow rebuilt it from scratch with
`ACME_EMAIL`/`MANAGE_DNS=true` and the new `hetzner`/cert-manager Secret, and
a canary `Certificate` went `Ready=True` via a real Let's Encrypt DNS-01
challenge through the `acme.hetzner.com`/`hetzner` webhook — confirmed by
`kubectl describe certificate` showing `Reason: Ready` /
`"The certificate has been successfully issued"`. This proves Steps 0.1,
0.2/0.5's local variant, and 0.3 all work correctly together on a live
cluster, and incidentally re-validated the fixed `destroy-cluster.sh`.

**✅ VERIFIED END-TO-END (2026-07-19), Hetzner/production target — but only
after a real outage.** Production's first boot hit exactly the gap flagged
below as the open risk: the overlay wasn't pinned past `v1.2.20` yet, so the
webhook Application never deployed and every Certificate sat stuck pending.
See Step 0.4 for the full incident writeup and the corrected pre-flight
order (bump the overlay tag *before* first boot, not after).

### Why first
The current `letsencrypt-prod` ClusterIssuer (generated by
`infrastructure/cloud-init/k3s-node.yaml.tpl`) uses an HTTP-01 solver, which
requires every hostname to answer on public port 80. Any later privatization
breaks renewals. DNS-01 solves challenges through the Hetzner DNS zone
(already managed by this repo via `hcloud_zone_rrset`), independent of
reachability. Hetzner ships an **official** cert-manager webhook for the new
Cloud DNS API: `hetzner/cert-manager-webhook-hetzner`
(Helm repo `https://charts.hetzner.cloud`, solver `groupName: acme.hetzner.com`,
`solverName: hetzner`, token Secret `hetzner`/`token` in the `cert-manager`
namespace).

### Step 0.1 — New ArgoCD Application for the webhook (IMPLEMENTED)
New file `infrastructure/kubernetes/apps/cert-manager-webhook-hetzner.yaml`,
modeled on `apps/cert-manager.yaml`:

- `sync-wave: "-10"` (foundational infra tier; the ArgoCD retry policy absorbs
  the ordering dependency on cert-manager within the wave).
- `repoURL: https://charts.hetzner.cloud`, `chart: cert-manager-webhook-hetzner`,
  pinned `targetRevision: 0.8.0` (confirmed latest via `helm search repo
  --versions` at implementation time).
- `namespace: cert-manager`. No `helm: values:` override needed — the chart's
  defaults already set `groupName: acme.hetzner.com`,
  `certManager.namespace: cert-manager`, and
  `certManager.serviceAccountName: cert-manager`, matching this repo's single
  cert-manager install exactly (checked via `helm show values`).

Registered in `infrastructure/kubernetes/kustomization.yaml` right after
`apps/cert-manager.yaml`.

### Step 0.2 — API token Secret (IMPLEMENTED, reusing the existing token)
No new terraform variable was needed. `var.hcloud_token` already exists in
both `infrastructure/terraform/variables.tf` and `terraform-staging/variables.tf`
("The Hetzner DNS API Token"), is already populated with a real,
DNS-zone-write-capable Hetzner Cloud API token on every apply
(`smallworlds-init.sh` writes it into `terraform.tfvars`;
`test-pr-locally.sh` exports `TF_VAR_hcloud_token`), and is already proven to
work for DNS automation — it's the same token used by `ensure_dns_zone()` in
`smallworlds-init.sh` and by the Stalwart mail DNS provisioner
(`stalwart-dns-secrets`). It was simply never threaded into the cloud-init
`templatefile()` call. Fixed that instead of inventing a parallel credential:

1. `infrastructure/terraform/main.tf`: added `hcloud_token = var.hcloud_token`
   to the `templatefile(...)` call.
2. `infrastructure/terraform-staging/main.tf`: added `hcloud_token = ""`
   (required because `templatefile()` needs every referenced variable bound
   regardless of which `%{ if }` branch is live — matches the existing
   `acme_email = ""` convention for staging).
3. `infrastructure/cloud-init/k3s-node.yaml.tpl`, inside the existing
   `%{ if acme_email != "" }` branch, added a second bootstrap manifest file
   `cert-manager-webhook-hetzner-secret.yaml` next to `letsencrypt-prod.yaml`:

   ```yaml
   apiVersion: v1
   kind: Secret
   metadata:
     name: hetzner
     namespace: cert-manager
   type: Opaque
   stringData:
     token: "${hcloud_token}"
   ```

   No explicit `Namespace: cert-manager` object was added. Precedent: the
   existing `letsencrypt-prod.yaml` ClusterIssuer already gets applied by
   k3s's built-in manifest controller before cert-manager's own CRDs exist
   (ArgoCD installs them later), and it already works — k3s's addon
   controller retries failed applies until they succeed. The same retry
   carries the Secret until the `cert-manager` Application creates its
   namespace via `CreateNamespace=true`.

   Verified by rendering the template standalone via `templatefile()` in a
   throwaway `terraform apply` (both the `acme_email != ""` and `== ""`
   branches), confirming no "vars map does not contain key" errors, and
   `yaml.safe_load`-parsing both the outer cloud-config and the two embedded
   heredoc manifests (ClusterIssuer, Secret).

### Step 0.3 — Switch the ClusterIssuer solver
In the same tpl, replace the `http01` solver of `letsencrypt-prod` with:

```yaml
solvers:
- dns01:
    webhook:
      groupName: acme.hetzner.com
      solverName: hetzner
      config:
        tokenSecretKeyRef:
          name: hetzner
          key: token
```

(Verify the exact `config` key names against
`docs/reference/issuer-configuration.md` in the webhook repo at implementation
time.)

The self-signed `%{ else }` branch (staging) is untouched.

### Step 0.4 — Live rollout (operator runbook, Hetzner — EXECUTED 2026-07-19, the hard way)
`ignore_changes = [user_data]` on `hcloud_server.smallworlds_pilot_node` means
the tpl edits do **not** touch the running VM; they only apply on the next
deliberate rebuild. On a *fresh* node, though, `ignore_changes` doesn't
matter — there is no prior `user_data` to ignore, so the ClusterIssuer
DNS-01 switch and the `hetzner` token Secret land automatically at first
boot via cloud-init. That part needs no manual step at all on a new cluster.

**What actually happened on the first production cutover:** production was
built via a plain `smallworlds-init.sh` Hetzner run. Cloud-init duly brought
up `letsencrypt-prod` already pointed at the `dns01`/`hetzner` webhook
solver — but the overlay (`my-community-config`) was still pinned to a tag
older than `v1.2.20` (the tag that added
`apps/cert-manager-webhook-hetzner.yaml`, Step 0.1), so ArgoCD never
deployed the webhook itself. Result: **every** Certificate in the cluster
sat `pending` on `hetzner.acme.hetzner.com is forbidden: ... cannot create
resource "hetzner"` — a total outage, no app reachable, with nothing in the
rollout-watch loop calling it out (all Applications the overlay *did* know
about came up `Synced`+`Healthy`; the missing one simply didn't exist to be
unhealthy). Fixed live by applying the webhook Application directly
(`kubectl apply -f infrastructure/kubernetes/apps/cert-manager-webhook-hetzner.yaml`
— self-contained, pulls straight from `https://charts.hetzner.cloud`, so it
doesn't need the overlay to work) and deleting the stuck `Challenge` objects
(`kubectl delete challenges --all -A`) to force an immediate retry once the
RBAC the chart installs was in place.

**Lesson for next time — and this generalizes beyond Phase 0:** whenever a
cloud-init change assumes an ArgoCD-deployed resource exists (as this
ClusterIssuer switch does), the overlay's pinned tag MUST already include
that resource *before* the node's first boot, not after. Cloud-init changes
in this repo go live immediately on a fresh node; ArgoCD-managed changes
only go live once the overlay catches up. A fresh bootstrap is exactly the
moment those two are most likely to be out of sync. Concretely: **before
running `smallworlds-init.sh` against a brand-new Hetzner (or first-time
internet-exposed local) node, bump the overlay's pinned tag to the latest
release first** (see README's "Managing Updates" section) — don't rely on
Renovate's weekly cadence to have caught up yet on a repo that was only
just created.

Superseded runbook (kept for reference — only relevant when hand-cutting an
*existing* running cluster over, e.g. non-Hetzner or `ignore_changes`
actually in effect):

1. Merge the repo changes; tag; bump the pinned ref in `my-community-config`;
   let ArgoCD deploy the webhook Application.
2. `kubectl apply` the `hetzner` token Secret and the updated
   `letsencrypt-prod` ClusterIssuer manually against production
   (`export KUBECONFIG=~/.smallworlds/kubeconfigs/production.yaml`).
3. **Canary before trusting it**: create a throwaway Certificate for a new
   hostname (e.g. `dns01-canary.<domain>` — add a temporary A record or just
   rely on DNS-01 not needing one) and watch it become `Ready`. Check the
   webhook pod logs for the TXT record round-trip.
4. Force-renew one real, low-risk certificate (e.g. the dashboard) with
   `cmctl renew` and confirm renewal via DNS-01. All other certs migrate
   automatically at their natural renewal (~30 days before expiry). The
   existing rate-limit protection (`admin-tools/restore-certs-from-laptop.sh`)
   is unaffected.

### Step 0.5 — Local-target mirror (IMPLEMENTED AND VERIFIED LIVE)
CLAUDE.md flags that `infrastructure/cloud-init/k3s-node.yaml.tpl` (Hetzner)
and `infrastructure/local/bootstrap-local-node.sh` (LAN/local target)
implement the same bootstrap contract independently and usually need mirrored
changes — Steps 0.1–0.4 above only touched the Hetzner side. Mirrored:

1. `infrastructure/local/bootstrap-local-node.sh`: same `dns01` webhook solver
   swap on the `letsencrypt-prod` ClusterIssuer. Bonus over Hetzner: DNS-01
   doesn't need port 80/443 reachability, so `ACME_EMAIL` can now be set on a
   NAT'd LAN machine with no port forwarding — updated that variable's doc
   comment accordingly (it used to warn HTTP-01 fails behind NAT).
2. `smallworlds-init.sh`: the local target has no `templatefile()` call to
   thread a token through — its bootstrap manifests come from the
   `SECRETS_FILE` heredoc this script generates and hands to
   `bootstrap-local-node.sh` (used for **both** deploy targets). Added a
   `cert-manager` Namespace + `hetzner` Secret block there, right after the
   existing `stalwart-dns-secrets` block, reusing the already-collected
   `$SECRETS_HCLOUD_TOKEN` — which is already correctly gated (blanked for
   LAN-only local installs, real value whenever `LOCAL_PUBLIC=yes` i.e.
   whenever `ACME_EMAIL` is also set, so no broken configured-solver/no-token
   state is reachable).

**⚠️ Rollout mechanics differ sharply from Steps 0.1–0.4, and this is easy to
get wrong:** `bootstrap-local-node.sh` and `smallworlds-init.sh` are **not**
part of the GitOps loop at all — ArgoCD never fetches them. They run exactly
once, standalone, over SSH, at initial node provisioning (or an explicit
manual re-run). Consequently:

- Bumping the pinned tag in a community overlay repo (`update-community-version.sh`)
  has **zero effect** on this fix, no matter how new the tag is — there is
  nothing here for ArgoCD to sync. Don't reach for that tool expecting it to
  help.
- The fix is **dormant on any already-running local-target cluster** until
  that node is rebuilt from scratch via a fresh `smallworlds-init.sh` run. A
  live cluster bootstrapped before this commit keeps its old HTTP-01
  `letsencrypt-prod` ClusterIssuer indefinitely unless either the node is
  reinstalled, or the operator manually reruns `bootstrap-local-node.sh`
  against it, or applies the same `kubectl apply` live-cutover documented in
  Step 0.4 by hand. (The `.dev` cluster itself was rebuilt on 2026-07-18 —
  see the verification note above — so this no longer applies to it
  specifically, only to any other pre-existing local-target install.)
- This is a real regression risk to watch for: if a local-target node is ever
  rebuilt from a commit *older* than this one (e.g. a pinned older release
  script cached somewhere), it silently reverts to HTTP-01.

### Rollback
Re-apply the previous ClusterIssuer with the `http01` solver (ports 80/443 are
still open). No cert is lost; issued certs live in Secrets regardless of
solver.

## Phase 1 — Headscale + close 22/6443

**Status: not started.** No Headscale tenant, ArgoCD Application, tailscaled
install, or firewall change exists yet — everything below is still a plan,
not a description of current state.

**Local-target scope question: RESOLVED — Phase 1 is Hetzner/production-only.**
This phase is written entirely against the Hetzner target — `hcloud_firewall`,
`terraform apply`, `root@<ip>` SSH, "the Hetzner Cloud console" as
break-glass. Phase 0 looked the same way at first and turned out to need a
full local-target mirror (Step 0.5); Phase 1 does not:

- **No public exposure to lock down.** `doc/local-deployment.md` documents
  the only required router port-forwards for an internet-exposed local
  install as `80/tcp, 443/tcp, 10000/udp` (confirmed against
  `smallworlds-init.sh`'s own `LOCAL_PUBLIC=yes` reminder message) — neither
  script ever instructs forwarding 22 or 6443. Confirmed directly with the
  operator (2026-07-18): the `.dev` box's router has no forward for either
  port beyond those three. SSH and the kube API are therefore already
  LAN-only on this deployment, by default and in practice — there is nothing
  for a VPN overlay to additionally close off.
- **The plumbing doesn't map anyway.** `admin-tools/lib/cluster-env.sh`
  already models `LOCAL_SSH_TARGET` as a non-root user + sudo, not
  `root@<ip>` (Step 1.3's assumption); there is no `hcloud_firewall` resource
  for a local node to edit (Step 1.4); "Hetzner Cloud console → VNC"
  break-glass (Step 1.5) has no local-target equivalent (physical/LAN access
  fills that role instead, and already exists for free).
- **Conclusion:** implement Phase 1 (Steps 1.1–1.5 below) against the Hetzner
  target only, as written, once a production node exists. Do not build a
  local-target mirror for it — revisit only if a future local-target
  operator's own router config changes this (forwards 22/6443 for personal
  remote-access convenience outside this project's instructions), which
  would be a per-operator decision to handle in their own README/runbook
  notes, not something this repo needs to account for by default.

**Sequencing constraint discovered while implementing — do not skip this
when picking Phase 1 back up:** the firewall lockdown (Step 1.4) must never
be part of the *same* `terraform apply` that first provisions a node. On a
brand-new node's very first apply, port 22 has to stay open — `smallworlds-init.sh`'s
Hetzner flow itself depends on SSH (waiting for `/etc/rancher/k3s/k3s.yaml`,
fetching the kubeconfig, reading the ArgoCD password) to finish bootstrapping
in the first place. The correct order once a production node exists: (1) let
it boot normally with 22 open, (2) `smallworlds-init.sh` completes, (3) wait
for the `headscale` Application to sync, (4) run `setup-vpn.sh`, (5) verify
tailnet SSH/kubectl actually work, (6) only then apply the firewall change
from Step 1.4 as its own separate `terraform apply`. Similarly, Step 1.3's
`cluster-env.sh` repoint should not unconditionally replace the public-IP
path — every other admin-tool (including `smallworlds-init.sh`'s own
bootstrap) still needs to reach a node before its tailnet identity exists,
so this needs to become a fallback/opt-in, not a hard switch.

### Architecture in one paragraph
Headscale (the coordination server) runs **in-cluster** as a normal GitOps
tenant behind Traefik at `vpn.<domain>` — it must stay on a public hostname so
clients can reach it. The VM itself joins the tailnet via `tailscaled` on the
host, giving the node a stable `100.64.0.0/10` address; sshd and the kube API
already bind all interfaces, so both become reachable over the tailnet with no
further plumbing. A public DNS record `admin.<domain>` points at the node's
tailnet IP (a CGNAT address in public DNS is fine — it only routes for tailnet
members) and becomes the canonical operator endpoint for SSH and kubectl.
Data plane note: established peers keep working even when the in-cluster
control plane is down (clients cache the netmap), so a broken cluster does not
immediately lock the operator out — and the Hetzner console is the final
break-glass.

### Step 1.1 — Headscale tenant (IMPLEMENTED, not yet deployed live)
New directory `infrastructure/kubernetes/tenants/headscale/`:

- **`deployment.yaml`** — image `headscale/headscale:<pinned version>` (pin by
  digest like the other tenants), running `headscale serve`.
  - Mount a config ConfigMap at `/etc/headscale/config.yaml`. Key settings:
    `server_url: https://vpn.<domain>`, `listen_addr: 0.0.0.0:8080`,
    `prefixes.v4: 100.64.0.0/10`, SQLite at `/var/lib/headscale/db.sqlite`,
    DERP: keep the default (Tailscale's public DERP map) for relay fallback —
    direct WireGuard will be the normal path since the server has a public IP.
    Disable/lengthen node key expiry for infrastructure nodes
    (`node_update_check_interval`, key expiry ≥ 180d) to reduce lockout risk.
  - **PVC** for `/var/lib/headscale` using the cluster default storage class
    (k3s local-path stores under `/var/lib/rancher/k3s/storage`, which is
    symlinked onto the persistent Hetzner volume — the node-key DB survives VM
    rebuilds).
- **`service.yaml` + `ingress.yaml`** — Ingress `vpn.<domain>` with
  `cert-manager.io/cluster-issuer: letsencrypt-prod` + `tls` block, per the
  README checklist. Headscale speaks HTTP + WebSocket/gRPC; Traefik handles
  this without special middleware.
- **`kustomization.yaml`** — standard tenant wiring. No Keycloak client and no
  Garage bucket needed in this phase (OIDC onboarding is Phase 2).

New file `infrastructure/kubernetes/apps/headscale.yaml`: ArgoCD Application,
**`sync-wave: "0"`** — it depends on Traefik and cert-manager (wave -10) but
not on Keycloak. Register in `infrastructure/kubernetes/kustomization.yaml`
and add `"vpn"` to the `app_records` for_each list in
`infrastructure/terraform/main.tf`.

### Step 1.2 — tailscaled on the host (repo-side IMPLEMENTED; live join = admin-tools/setup-vpn.sh, UNTESTED)
Two halves, matching the Phase 0 pattern (repo change for rebuilds + live
operator step now):

- **`infrastructure/cloud-init/k3s-node.yaml.tpl`**: install the tailscale
  package (`curl -fsSL https://tailscale.com/install.sh | sh` in `runcmd`, or
  the apt repo explicitly). Install only — joining needs a preauth key that
  only exists after ArgoCD has deployed Headscale, so joining stays an
  admin-tool step. Also consider adding the package to
  `infrastructure/golden-image/` so staging/rebuild boots don't re-download it.
- **New script `admin-tools/setup-vpn.sh`** (operator runbook as code):
  1. `kubectl -n headscale exec deploy/headscale -- headscale users create infra`
  2. `... headscale preauthkeys create --user infra --expiration 1h`
  3. `ssh root@<server> "tailscale up --login-server=https://vpn.<domain> --auth-key=<key>"`
  4. Read the node's tailnet IP (`tailscale ip -4`), print it, and create/update
     the `admin.<domain>` DNS record pointing at it (via `hcloud` CLI or a
     `tailnet_admin_ip` terraform variable — CLI is simpler since the IP is
     only known post-join).
  5. Append `tls-san: ["admin.<domain>", "<tailnet-ip>"]` to
     `/etc/rancher/k3s/config.yaml` on the server and `systemctl restart k3s`
     — without this the kube API cert is invalid when addressed via the
     tailnet. (Also add this line to the cloud-init tpl guarded by a new
     template var so rebuilds keep it.)
  6. Enroll the operator laptop: `tailscale up --login-server=https://vpn.<domain>`
     + `headscale nodes register` with the printed machine key.
