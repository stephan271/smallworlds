# Implementation Plan: Private Overlay Network — Phases 0 & 1

Goal of the overall initiative: make the end-user apps reachable only inside a
private community overlay network (Headscale/Tailscale), while mail, Jitsi
(guests) and Keycloak stay public. This document covers the two preparatory
phases only:

- **Phase 0** — switch Let's Encrypt validation from HTTP-01 to DNS-01, so
  hostnames no longer need to be publicly reachable to get/renew certificates.
- **Phase 1** — deploy Headscale as a tenant, join the VM to the tailnet, and
  close SSH (22) and the Kubernetes API (6443) to the public internet.

Neither phase changes anything a community member can see. Both are pure
operator-facing hardening.

Decisions already made:
- Overlay: **Headscale** (self-hosted, single binary, official Tailscale clients).
- SSH policy: **port 22 fully closed** on production once VPN access is verified.
  Break-glass is the Hetzner Cloud console (VNC) plus temporarily re-adding a
  firewall rule there.
- Staging (`terraform-staging/`) stays fully public and unchanged — the PR
  validation pipeline (`test-pr-locally.sh`, e2e suite) must keep working
  without a VPN.

## Phase 0 — DNS-01 certificate validation

### Why first
The current `letsencrypt-prod` ClusterIssuer (generated by
`infrastructure/cloud-init/k3s-node.yaml.tpl`) uses an HTTP-01 solver, which
requires every hostname to answer on public port 80. Any later privatization
breaks renewals. DNS-01 solves challenges through the Hetzner DNS zone
(already managed by this repo via `hcloud_zone_rrset`), independent of
reachability. Hetzner ships an **official** cert-manager webhook for the new
Cloud DNS API: `hetzner/cert-manager-webhook-hetzner`
(Helm repo `https://charts.hetzner.cloud`, solver `groupName: acme.hetzner.com`,
`solverName: hetzner`, token Secret `hetzner`/`token` in the `cert-manager`
namespace).

### Step 0.1 — New ArgoCD Application for the webhook
New file `infrastructure/kubernetes/apps/cert-manager-webhook-hetzner.yaml`,
modeled on `apps/cert-manager.yaml`:

- `sync-wave: "-10"` (foundational infra tier; the ArgoCD retry policy absorbs
  the ordering dependency on cert-manager within the wave).
- `repoURL: https://charts.hetzner.cloud`, `chart: cert-manager-webhook-hetzner`,
  pinned `targetRevision` (latest release, v0.8.x at time of writing).
- `namespace: cert-manager`.

Register it in `infrastructure/kubernetes/kustomization.yaml`.

### Step 0.2 — API token Secret
The webhook needs a Hetzner Cloud API token (Read & Write on the project
holding the DNS zone). Follow the existing bootstrap-manifest pattern rather
than inventing a new secret channel:

1. Add `variable "dns_api_token" { sensitive = true }` to
   `infrastructure/terraform/variables.tf` (and to the operator's
   `terraform.tfvars`). Use a **dedicated token**, not the main
   `hcloud_token`, so it can be rotated independently.
2. Pass it into `templatefile(...)` in `infrastructure/terraform/main.tf`
   (`user_data` block) and in `terraform-staging/main.tf` pass `""`.
3. In `infrastructure/cloud-init/k3s-node.yaml.tpl`, inside the existing
   `%{ if acme_email != "" }` branch, write a second bootstrap manifest next to
   `letsencrypt-prod.yaml`:

   ```yaml
   apiVersion: v1
   kind: Secret
   metadata:
     name: hetzner
     namespace: cert-manager
   stringData:
     token: "${dns_api_token}"
   ```

   Note: the `cert-manager` namespace must exist before k3s applies this;
   add a bare Namespace object in the same manifest file (k3s applies
   multi-doc YAML in order).

### Step 0.3 — Switch the ClusterIssuer solver
In the same tpl, replace the `http01` solver of `letsencrypt-prod` with:

```yaml
solvers:
- dns01:
    webhook:
      groupName: acme.hetzner.com
      solverName: hetzner
      config:
        tokenSecretKeyRef:
          name: hetzner
          key: token
```

(Verify the exact `config` key names against
`docs/reference/issuer-configuration.md` in the webhook repo at implementation
time.)

The self-signed `%{ else }` branch (staging) is untouched.

### Step 0.4 — Live rollout (operator runbook)
`ignore_changes = [user_data]` on `hcloud_server.smallworlds_pilot_node` means
the tpl edits do **not** touch the running VM; they only apply on the next
deliberate rebuild. So the change must additionally be applied live, once:

1. Merge the repo changes; tag; bump the pinned ref in `my-community-config`;
   let ArgoCD deploy the webhook Application.
2. `kubectl apply` the `hetzner` token Secret and the updated
   `letsencrypt-prod` ClusterIssuer manually against production
   (`export KUBECONFIG=~/.smallworlds/kubeconfigs/production.yaml`).
3. **Canary before trusting it**: create a throwaway Certificate for a new
   hostname (e.g. `dns01-canary.<domain>` — add a temporary A record or just
   rely on DNS-01 not needing one) and watch it become `Ready`. Check the
   webhook pod logs for the TXT record round-trip.
4. Force-renew one real, low-risk certificate (e.g. the dashboard) with
   `cmctl renew` and confirm renewal via DNS-01. All other certs migrate
   automatically at their natural renewal (~30 days before expiry). The
   existing rate-limit protection (`admin-tools/restore-certs-from-laptop.sh`)
   is unaffected.

### Rollback
Re-apply the previous ClusterIssuer with the `http01` solver (ports 80/443 are
still open). No cert is lost; issued certs live in Secrets regardless of
solver.

## Phase 1 — Headscale + close 22/6443

### Architecture in one paragraph
Headscale (the coordination server) runs **in-cluster** as a normal GitOps
tenant behind Traefik at `vpn.<domain>` — it must stay on a public hostname so
clients can reach it. The VM itself joins the tailnet via `tailscaled` on the
host, giving the node a stable `100.64.0.0/10` address; sshd and the kube API
already bind all interfaces, so both become reachable over the tailnet with no
further plumbing. A public DNS record `admin.<domain>` points at the node's
tailnet IP (a CGNAT address in public DNS is fine — it only routes for tailnet
members) and becomes the canonical operator endpoint for SSH and kubectl.
Data plane note: established peers keep working even when the in-cluster
control plane is down (clients cache the netmap), so a broken cluster does not
immediately lock the operator out — and the Hetzner console is the final
break-glass.

### Step 1.1 — Headscale tenant
New directory `infrastructure/kubernetes/tenants/headscale/`:

- **`deployment.yaml`** — image `headscale/headscale:<pinned version>` (pin by
  digest like the other tenants), running `headscale serve`.
  - Mount a config ConfigMap at `/etc/headscale/config.yaml`. Key settings:
    `server_url: https://vpn.<domain>`, `listen_addr: 0.0.0.0:8080`,
    `prefixes.v4: 100.64.0.0/10`, SQLite at `/var/lib/headscale/db.sqlite`,
    DERP: keep the default (Tailscale's public DERP map) for relay fallback —
    direct WireGuard will be the normal path since the server has a public IP.
    Disable/lengthen node key expiry for infrastructure nodes
    (`node_update_check_interval`, key expiry ≥ 180d) to reduce lockout risk.
  - **PVC** for `/var/lib/headscale` using the cluster default storage class
    (k3s local-path stores under `/var/lib/rancher/k3s/storage`, which is
    symlinked onto the persistent Hetzner volume — the node-key DB survives VM
    rebuilds).
- **`service.yaml` + `ingress.yaml`** — Ingress `vpn.<domain>` with
  `cert-manager.io/cluster-issuer: letsencrypt-prod` + `tls` block, per the
  README checklist. Headscale speaks HTTP + WebSocket/gRPC; Traefik handles
  this without special middleware.
- **`kustomization.yaml`** — standard tenant wiring. No Keycloak client and no
  Garage bucket needed in this phase (OIDC onboarding is Phase 2).

New file `infrastructure/kubernetes/apps/headscale.yaml`: ArgoCD Application,
**`sync-wave: "0"`** — it depends on Traefik and cert-manager (wave -10) but
not on Keycloak. Register in `infrastructure/kubernetes/kustomization.yaml`
and add `"vpn"` to the `app_records` for_each list in
`infrastructure/terraform/main.tf`.

### Step 1.2 — tailscaled on the host
Two halves, matching the Phase 0 pattern (repo change for rebuilds + live
operator step now):

- **`infrastructure/cloud-init/k3s-node.yaml.tpl`**: install the tailscale
  package (`curl -fsSL https://tailscale.com/install.sh | sh` in `runcmd`, or
  the apt repo explicitly). Install only — joining needs a preauth key that
  only exists after ArgoCD has deployed Headscale, so joining stays an
  admin-tool step. Also consider adding the package to
  `infrastructure/golden-image/` so staging/rebuild boots don't re-download it.
- **New script `admin-tools/setup-vpn.sh`** (operator runbook as code):
  1. `kubectl -n headscale exec deploy/headscale -- headscale users create infra`
  2. `... headscale preauthkeys create --user infra --expiration 1h`
  3. `ssh root@<server> "tailscale up --login-server=https://vpn.<domain> --auth-key=<key>"`
  4. Read the node's tailnet IP (`tailscale ip -4`), print it, and create/update
     the `admin.<domain>` DNS record pointing at it (via `hcloud` CLI or a
     `tailnet_admin_ip` terraform variable — CLI is simpler since the IP is
     only known post-join).
  5. Append `tls-san: ["admin.<domain>", "<tailnet-ip>"]` to
     `/etc/rancher/k3s/config.yaml` on the server and `systemctl restart k3s`
     — without this the kube API cert is invalid when addressed via the
     tailnet. (Also add this line to the cloud-init tpl guarded by a new
     template var so rebuilds keep it.)
  6. Enroll the operator laptop: `tailscale up --login-server=https://vpn.<domain>`
     + `headscale nodes register` with the printed machine key.

### Step 1.3 — Repoint operator tooling
- `admin-tools/lib/cluster-env.sh`: `fetch_kubeconfig` rewrites the API
  endpoint to `$server` — for production/dev, `$server` must become
  `admin.<domain>` (or the tailnet IP) instead of the public IP. Add the
  address indirection where the server address is derived (likely
  `detect_*`/terraform-output helpers in the same file); staging keeps using
  the public IP.
- Verify over the tailnet **before** touching the firewall:
  `ssh root@admin.<domain>` works, and
  `KUBECONFIG=~/.smallworlds/kubeconfigs/production.yaml kubectl get nodes`
  works with the endpoint rewritten to `admin.<domain>`.

### Step 1.4 — Firewall lockdown (`infrastructure/terraform/main.tf`)
Only after 1.3 is verified. In `hcloud_firewall.k8s_firewall`:

- **Delete** the port `22` rule.
- **Delete** the port `6443` rule.
- **Add** `udp/41641` from `0.0.0.0/0, ::/0` (tailscaled's WireGuard port) so
  clients get direct connections instead of falling back to DERP relays.

`terraform apply` — firewall-only change, no VM replacement (and `user_data`
is under `ignore_changes` anyway). **Do not** mirror this in
`terraform-staging/main.tf`; staging keeps 22/6443 open by design.

### Step 1.5 — Post-lockdown verification
1. From a machine **on** the tailnet: SSH + kubectl work via `admin.<domain>`.
2. From a machine **off** the tailnet: `nc -zv <public-ip> 22` and `:6443`
   time out; `https://<apps>` all still load; mail ports still open.
3. Reboot the server once (Hetzner console): tailscaled rejoins on boot,
   SSH over tailnet comes back without manual steps.
4. Run the production e2e smoke tests (`./run-smoke-tests.sh <domain>`) —
   apps are still public in this phase, so the suite must pass unchanged.
5. Document break-glass in the README operator section: Hetzner Cloud console
   → VNC login, or temporarily re-add a `22` firewall rule scoped to the
   operator's current IP in the Hetzner UI.

### Rollback
Re-add the 22/6443 firewall rules in `main.tf` and apply (or ad-hoc in the
Hetzner console). Headscale tenant can stay deployed; nothing else depends on
it yet.

## Order of work / PR breakdown

| PR | Content | Risk |
|----|---------|------|
| 1 | Phase 0: webhook Application, token plumbing (tfvars → cloud-init → Secret), ClusterIssuer solver switch in tpl | Low — HTTP-01 stays available for rollback |
| 2 | Phase 1a: headscale tenant + Application + `vpn` DNS record + tailscale install in cloud-init + `setup-vpn.sh` | Low — additive only |
| 3 | Phase 1b: `cluster-env.sh` repoint + firewall lockdown (22/6443 out, 41641/udp in) + README break-glass docs | Medium — apply only after live tailnet verification |

Each PR validates via `./admin-tools/test-pr-locally.sh <branch>` as usual;
note that for PR 3 staging proves nothing about the lockdown itself (staging
firewall unchanged) — its verification is the manual runbook in 1.3/1.5.

## Explicit non-goals (later phases)
- No member onboarding, no Keycloak-OIDC join flow for Headscale (Phase 2).
- No app privatization, no `ipAllowList` middleware, no DNS repointing of app
  records (Phase 2).
- No Headscale ACL policy yet — the tailnet contains only operator devices and
  the node; default allow-all within the tailnet is acceptable until members
  join.
